# Strict Code-Quality Rules for AI PR Agent

## 1. Style Rules (derived from existing codebase)
- **Python version**: Target Python 3.10+ syntax (use `str | None`, `match-case`, `f-strings`).
- **Type hints**: Mandatory on all public functions, methods, and class attributes.
- **Naming**: `snake_case` for functions/variables, `PascalCase` for classes, `UPPER_CASE` for constants. Follow existing module conventions (`main.py`, `models.py`, `schemas.py`).
- **Imports**: Standard library → third‑party → local, each group separated by a blank line. Never use `import *`.
- **Docstrings**: Google‑style for all functions. Include parameters, return types, and raised exceptions.
- **Async**: Use `async def` for all FastAPI route handlers that perform I/O.
- **Formatting**: Automatic `black` (line length 88) and `isort` must pass; commit only after formatting.
- **Pydantic & FastAPI**: Use Pydantic V2 models (`model_validate`, `model_dump`) for request/response schemas. Re‑export schemas from `schemas.py`. Use `HTTPException` for client errors.

## 2. Required Practices
- **Tests**: Every PR must add or update tests in `test_*` files using `pytest`. Cover success, validation errors, edge cases, and database rollback.
- **Error Handling**: Always raise `HTTPException` with clear `status_code` and `detail`. Catch database integrity errors and re‑raise as 409 CONFLICT. Never expose raw tracebacks to the client.
- **Logging**: Use `logging.getLogger(__name__)`. Log at `INFO` level for every CUD operation; `ERROR` on exceptions. Format: `"Action: %s, Product ID: %s, User: %s"`. No `print()`.

## 3. Forbidden Patterns
- **No `print()`** for logging or debugging.
- **No bare `except:`** – always catch specific exceptions.
- **No mutable default arguments** (e.g., `def foo(x=[])`).
- **No blocking I/O** inside async routes (e.g., synchronous SQLAlchemy calls).
- **No raw SQL** without parameterization.
- **No secrets** hardcoded (use environment variables).
- **No global state** that is not explicitly initialized at startup.
- **No commented‑out code** left behind – delete instead.

## 4. Pull‑Request Checklist
Before `submit_pr`, ensure **all** these are true:
- [ ] All existing and new tests pass (`pytest -v`).
- [ ] Code is formatted (`black . && isort .`).
- [ ] No linting errors (flake8/pylint if configured).
- [ ] Every function/class has type hints and docstrings.
- [ ] Error handling covers at least: missing resource (404), validation error (422), database conflict (409).
- [ ] Logging statements added for every non‑trivial action.
- [ ] No debug prints, `# TODO` without ticket, or commented‑out code.
- [ ] Sensitive data never logged.
- [ ] Static files (`static/`) are minified and consistent with existing patterns.
- [ ] README updated if API behaviour or dependencies changed.
