## Code-Quality Rules for PR Submission

### Language & Framework Style Rules
- **Python style** follows PEP 8: 4‑space indentation, max line length 88 (Black‑compatible), snake_case for variables/functions, PascalCase for classes.
- **Type hints** are mandatory on all public functions and methods; use `Optional[X]` not `X | None` for consistency.
- **FastAPI** endpoints are written as `def` (synchronous) unless async logic is required. Endpoints must have `response_model` and `status_code` set explicitly for OpenAPI documentation.
- **SQLAlchemy** uses declarative mapping with `Base` in `models.py`. All database interactions go through `Session` (no raw SQL). Model attributes match the schema exactly.
- **Pydantic** schemas in `schemas.py` correspond to request/response bodies; use `ConfigDict(from_attributes = True)` for ORM mode.

### Required Practices
- **Testing** – Every endpoint must have at least two tests (happy path + error) in `test_*.py` using `TestClient` from FastAPI. Use a dedicated test database (SQLite in‑memory). Mock external dependencies only.
- **Error Handling** – Raise `HTTPException` with appropriate status codes (404, 422, 500). Catch `IntegrityError` and `DataError` from SQLAlchemy and convert them to user‑friendly HTTP errors. Never let unhandled exceptions propagate.
- **Logging** – Use Python’s `logging` module: `logger = logging.getLogger(__name__)`. Log all incoming requests (DEBUG), successful creations/updates (INFO), and all exceptions (ERROR). No `print()` statements.

### Forbidden Patterns
- Raw SQL strings (use ORM methods).
- Bare `except:` or `except BaseException:` – catch specific exceptions.
- Hard‑coded credentials, API keys, or secrets.
- Global mutable state (e.g., module‑level lists).
- Commented‑out code, dead code, or unused imports.
- Direct file I/O that the application does not control (use dependency injection for paths if needed).

### PR Checklist
Before calling `submit_pr`, verify every point:
- [ ] All new functionality is covered by tests; existing tests pass.
- [ ] Code coverage does not decrease (target ≥90%).
- [ ] Static analysis (Ruff/Flake8) reports zero issues.
- [ ] Endpoints are documented via FastAPI’s automatic schema (no missing `description` or `response_model`).
- [ ] No new dependencies added without prior discussion; if added, `requirements.txt` is updated and pinned.
- [ ] Error responses follow a consistent format (e.g., `{“detail”: “message”}`).
- [ ] Logging is added for all critical paths (creation, deletion, and any 4xx/5xx returns).
- [ ] No leftover debug code or print statements.
