# Architecture Reference: Grocery Store API

- **Server**: Uvicorn (0.24.0)
- **ORM**: SQLAlchemy (2.0.23)

- **Database**: SQLite (file `grocery_store.db`)
- **Testing**: Pytest (7.4.3) + HTTPX (0.25.2)

- **Frontend**: `static/index.html` – a standalone UI that consumes the API via JavaScript.

## Notable Patterns / Conventions
- **CRUD organisation**: the README explicitly describes a CRUD interface for grocery products. The code separates concerns: `models.py` holds SQLAlchemy table definitions, `schemas.py` contains Pydantic schemas for request/response validation, and `main.py` implements the route handlers using FastAPI’s decorators.
- **SQLite persistence**: a single file `grocery_store.db` is used as the database; it is likely created/configured in `models.py` (engine and session setup).
- **Partial updates**: the `PUT /products/{id}` endpoint supports partial updates (per the README), likely achieved with `response_model` and `exclude_unset` in Pydantic.
- **Filtering**: the list endpoint accepts query parameters for category and availability, implemented directly in the route handler.
- **Static serving**: FastAPI can mount a `StaticFiles` instance to serve the frontend; this is probably done in `main.py`.
- **Testing**: integration tests use a synchronous FastAPI test client (relying on `httpx`), and unit tests isolate the `hello` module. Database state during tests is unknown but may use an in‑memory SQLite fixture.
